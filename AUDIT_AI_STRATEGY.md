## 2. Auditing the Auditor (Chain Integrity Protection)

The current hash check in `server.ts` or `services/audit.ts` is only as good as the trust in the binary running it.

### A. The Separation of Concerns (Watchdog Strategy)

* **Independent Verifier**: Implement a standalone Go or Rust CLI tool (The "Watchdog") that reads the DB and verifies the hashes. This tool should be distributed as a signed binary, separate from the primary application.
* **Client-Side Verification**: In a desktop (Electron) context, give users an "Inspect Ledger" button that re-verifies the chain locally, making it harder for a central server (in SaaS mode) to spoof data.

### B. Transparency via Periodic Anchoring

* **Off-Server Anchor**: Every 24 hours (or at period closing), the latest hash tip should be sent to an immutable service (e.g., a "Hash Vault," a public blockchain, or a simple Amazon QLDB).
* **Tamper Evidence**: If the server is modified to skip hashes, the external anchor will no longer match the next day's report, providing irrefutable proof of tampering ("Proof of Ledger State").

### C. Checksum Execution Verification

* **Code Integrity**: Use Bun's internal binary bundling to sign the executable. Compare the binary checksum at startup against a known, remote version.
