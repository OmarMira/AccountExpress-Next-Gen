# Strategy for Error Reporting & Scalability — AccountExpress Next Gen

## 1. Centralized Error Reporting Strategy
Moving beyond `console.error` is critical for a production financial system.

### A. Backend (Bun/Elysia)
*   **Structured Logging**: Implement **Pino** or **Winston**. Logs should be JSON-formatted to facilitate ingestion by tools like ELK Stack or CloudWatch.
*   **Error Hooks**: Utilize Elysia's `.onError()` hook to:
    *   Categorize errors (Operational vs. Programmer).
    *   Sanitize messages (avoid leaking SQL or internal paths to the client).
    *   Assign a **Correlation ID** to every request/error for easy cross-referencing between client and server logs.
*   **External Sink**: Integrate **Sentry** for real-time alerting on exceptions.

### B. Frontend / Desktop (React/Electron)
*   **Global Boundary**: React `ErrorBoundary` at the root and major module levels.
*   **Local Persistence**: In the Electron context, stream logs to a local rotating file using `electron-log`. This allows users to "Export Debug Logs" for support tickets.

## 2. Ledger Scalability Strategy (Target: 1M+ Entries)
The current single-SQL-join architecture is excellent for thousands of rows, but will reach limits at scale.

### A. Database Partitioning
*   **By Company**: Since data is strict-multi-tenant, we can use **PostgreSQL Declarative Partitioning** by `company_id`. This keeps indexes small and queries fast.
*   **By Period**: Secondary partitioning by `fiscal_year` or `period_id` for older data.

### B. Materialized Balances & Closing
*   **Balance Summary Table**: Instead of calculating balances on-the-fly for every dashboard load, maintain an `account_balances` table updated via triggers or application logic whenever a journal entry is `posted`.
*   **Closing Periods**: When a period is `locked`, generate a "Closing Entry" that carries forward balances as "Opening Balances" for the next period. This allows the system to ignore entries from locked years during standard P&L calculations.

### C. Ledger Compression (Cold Storage)
*   For audit logs and old entries (>3 years), move them to cold storage (historical tables) or use columnar storage extensions like **TimescaleDB** or **Hydra** if massive analytics (BI) are needed.

## 3. Infrastructure & Deployment
*   **Stateless Backend**: Current Bun backend is already stateless. Scalability can be achieved via horizontal scaling (K8s) if the desktop-only scope expands to a SaaS model.
*   **Connection Pooling**: Use **PgBouncer** to manage a large number of concurrent connections efficiently.
